Object Oriented Paradigm 8 - Encapsulation Extension

Access Modifiers

Individual programming languages can help you implement encapsulation further by the use of access modifiers. These will specify which properties (attributes and methods) can be accessed from outside the class.

Recall that we provide getters, setters, and other methods to work with an object's data. Most of these methods will be specified as public. This means that another part of the system can call them (pass a message to them). So, during game play, when a player wins a hand, the player's set_score method will be used to modify that player's score. The score attribute will have been specified as private. This will prevent it from being accessed directly.

Where a programming language supports access modifiers, you would usually make all attributes private (so that data cannot be manipulated directly), and make most methods public (so that they can be invoked where access to data is needed). Methods that are used within the class (maybe to break down complex pieces of code) can be specified as private so that they can't be called by other objects.

There is a third access modifier: protected. You will learn about this when you look at the key OOP concept of inheritance.

Encapsulation as a form of data hiding

Encapsulation is a form of data hiding. The inner details and workings of the class do not need to be understood by anyone other than the class designer. Classes are often provided as part of software libraries for programmers to make use of. Each library will have documentation that will tell you the names of the classes and the (public) methods that you can use.

For example, consider a software library with prewritten classes that can produce simple charts and graphs.

Using this most basic documentation, you will be able to produce a pie chart.

chart = new PieChart()
chart.setLabels('Chocolate', 'Carrot', 'Coffee', 'Vanilla', 'Red velvet')
chart.setValues(12, 2, 3, 5, 16)
chart.setColours('Yellow', 'Blue', 'Purple', 'Green', 'Red')
chart.setTitle('Favourite Cakes')

You do not need to see the class's implementation to make use of the class. You don't need to know the names or data types of the attributes, nor the names of any private methods, nor how any of the methods are coded.

Interaction between objects

Objects interact by 'message passing'. This just means that they call each other's public methods. At this stage, you have not yet designed the class to represent a hand of cards, but imagine that you have a class called Hand and that it encapsulates a public add method (so that cards can be added to a hand).

The pseudocode below shows how a card could be dealt from a deck and added to a hand.

1. game_deck = Deck()
2. player_hand = Hand()
3. dealt_card = game_deck.deal(1)
4. player_hand.add(dealt_card)

Line 1: creates an instance of a card deck (complete with 52 cards)

Line 2: creates an instance of a player hand

Line 3: uses the deal method of the Deck class to remove a card from the deck

Line 4: uses the add method of the Hand class to add the card to the player's hand

Lines 3 and 4 could be combined to better demonstrate the interaction:

player_hand.add(game_deck.deal(1))

Summary

  • Encapsulation means protecting data by bundling the data and the methods that work on that data within a class.
  • The attributes should be declared as private, which means that they are hidden within the class.
  • Public methods must be provided to access or modify attributes (as required). These methods are known, colloquially, as getters and setters.
  • Accessors and mutators are the formal names for getters and setters.
  • Objects interact by passing messages.
  • Designing classes to be stand-alone allows class designers to change the implementation details without any adverse effect on the applications that make use of them.
